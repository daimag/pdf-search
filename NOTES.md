# 設計・運用メモ（Q&A / 決定ログ）

見積PDF検索ツールについて、検討の過程で出た「なぜこうしたか」「よくある疑問」をまとめたもの。
セットアップ手順は [SETUP-SERVER.md](SETUP-SERVER.md)、引き継ぎは [HANDOFF.md](HANDOFF.md) を参照。

---

## このツールは何をするか

NAS上の見積PDF（`\\nas13b87a\daimag_SHR\0600.見積PDF`、約495件、3フォーマット）を
**日付・商品名・仕入先・見積番号など**でブラウザ検索し、その場でPDFを表示する。

---

## アーキテクチャと判断理由

- **抽出とWebを分離**（索引作成バッチ ＋ 検索Webサイト）
  検索のたびにNASを読まない設計。検索は一瞬、NASが落ちても検索は動く。
- **抽出エンジン = PyMuPDF(fitz)**
  poppler/pdftotext は Adobe-Japan1(90ms-RKSJ-H) CMap不足で日本語が抜けない → 不採用。
- **検索 = LIKE部分一致**（NFKC正規化＋小文字）
  FTS5 trigramは3文字未満クエリが空振り → 495件ならLIKEで十分高速。
- **Web = Next.js + node:sqlite（Node24組み込み）**
  better-sqlite3はNode24用prebuilt無く回避。**Node.js 24以上が必須**。
- **index.db はマシン間で使い回せる**（中はNASのUNCパス。同じLANならどのPCでも同じPDFを開ける）。

## よくある疑問（このセッションのQ&A）

### Q. PDFのアップロードは必要？
**不要。** PDFはNASに置く運用のまま。アプリに何かを上げる手間はゼロ。

### Q. 常にNASを見てるの？
**検索時は見ない。** NASを触るのは2回だけ：
1. **索引作成のとき**（まとめて1回。update-index or 夜間自動）
2. **PDFを開くとき**（そのファイル1個だけ）
検索そのものはSQLite(index.db)を見るだけ＝爆速。NASが落ちていても検索は動く。

### Q. 索引なし（バッチ無し）で495件はキツい？
- **全文検索を毎回ライブでやると約2分/回** → 実用外。
- ファイル名・仕入先・日付だけで良ければ索引なし(ライブ)も可能だが、NAS常時接続が前提。
- 全文検索を続けるなら**索引は必須**。バッチは「夜間自動＋"今すぐ更新"ボタン」で意識せず運用できる。
  （差分更新なので2回目以降は一瞬。全2分は初回だけ）

### Q. `pnpm dev` / `npm run dev` で何が挙がってる？ Web仮想？
- **Node.jsのプロセスが1個**立ち上がり、ポート(3100)で待ち受けているだけ。
- **仮想化ではない**（VM/Docker/IISの仮想サイトではない）。そのnode.exe自体がWebサーバー。
- `netstat -ano | findstr :3100` でポートを握るPIDを特定できる。

### Q. dev と start の違い
- `dev`（`next dev`）= **開発用**。保存ごとに再コンパイル＋ホットリロード。やや重い。
- `start`（`next start`）= **本番用**。ビルド済みを配信。速い。事前に `next build` が必要。
- **本番サーバーでは dev を使わず build→start（またはNSSMサービス化）。**

### Q. IISを立てると何が変わる？ Nodeポートは使う？
- **IISはNodeの代わりではなく"玄関(80番)"。Nodeは"裏(3100)"で今まで通り動く。両方必要。**
- IISが `/pdfsearch` を `http://localhost:3100` へリバースプロキシ転送する。
  → **Nodeポートは内部通信用として使い続ける**（外向けには閉じられる）。
- 利用者URLは `http://server:3100` → `http://server/pdfsearch`（ポート番号が消える）。
- 必要: IISの `ARR` ＋ `URL Rewrite`（無料）／ Next.jsの `basePath` 設定＋再ビルド／ Nodeのサービス化。
- iisnode方式もあるが古くNext.jsと相性が悪いので非推奨。

## 運用の注意

- **サービス化(NSSM)推奨**だが、**NASにアクセスできるユーザーで実行**すること（SYSTEMだとNAS共有が見えないことがある）。
- **本番フォルダと開発(Cursor)フォルダは分ける**（同一PCでも）。編集中の状態が公開される事故を防ぐ。
- **2環境の同期はGitHub経由**：開発したら `git push`、もう片方は作業前に `git pull`。
  `data/index.db` はGit管理外なので各マシンで生成 or コピー。
